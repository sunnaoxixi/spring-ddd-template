### DDD（领域驱动设计）
- 定义：Domain-Driven Design 的缩写，本项目采用的核心架构方法论。强调以业务领域为中心进行系统设计，通过聚合根、实体、值对象、领域服务等概念建模复杂业务逻辑，实现技术实现与业务规则的解耦。项目严格遵循六边形架构分层：adaptor → application → domain → infrastructure。
- 别名：领域驱动设计、domain-driven-design

### RBAC（基于角色的访问控制）
- 定义：Role-Based Access Control 的缩写，项目采用的权限模型。通过角色（role）和权限点（permission）两级结构实现细粒度权限控制，权限点命名规则为 `{模块}:{read|write}`，用户通过拥有角色间接获得权限集合。内置 admin 和 user 两种基础角色，支持自定义角色和权限分配。
- 别名：角色权限模型、rbac

### 聚合根（Aggregate Root）
- 定义：DDD 中的核心概念，指一组相关对象的聚集，作为外部访问的唯一入口。聚合根负责维护内部一致性，所有业务逻辑都封装在聚合根或其内部实体方法中。项目中每个业务领域都有一个对应的聚合根类，如 UserAggregate、RoleAggregate 等。
- 别名：aggregate-root、聚合根

### 值对象（Value Object）
- 定义：DDD 中的不可变对象，用于封装具有相等性语义的数据。值对象没有唯一标识，比较基于属性值而非引用地址。项目中值对象类名必须以 Value 结尾，放置在 `model/value/` 目录下，继承 BaseValue 基类。用于封装如 PermissionKeysValue 等业务概念。
- 别名：value-object、vo

### 写模式
- 定义：DDD 四种开发模式之一，用于处理修改数据状态的业务场景。标准流程：领域服务先获取分布式锁 → 加载聚合根 → 调用聚合根业务方法 → repository.save() → finally 释放锁。适用于订单创建、状态变更等需要修改数据的业务。
- 别名：写操作模式、write-pattern

### 读模式
- 定义：DDD 四种开发模式之一，用于纯查询场景。聚合根仅作为数据载体，不包含业务逻辑，主要进行数据转换和组装。适用于报表查询、详情展示等只读操作。
- 别名：读操作模式、read-pattern

### 权限点
- 定义：RBAC 模型中最细粒度的权限单元，命名规则为 `{模块}:{操作}`，如 `system:user:read`、`system:user:write`。系统采用权限点+角色的两级授权机制，用户通过角色间接获得权限点集合。当前接口 `/api/auth/me` 仅返回角色列表，不直接返回权限点集合。
- 别名：permission-key、perm-key

### 操作日志
- 定义：系统审计日志的一种，记录用户对系统的各种操作行为。通过 `@OperLog(module, action)` 注解自动采集，包含操作人、业务模块、操作动作、请求地址、参数摘要、结果、耗时、IP 等信息。异步落库，避免影响主业务流程性能。
- 别名：oper-log、operation-log

### 登录日志
- 定义：系统审计日志的另一种，专门记录用户的登录行为。包含用户 ID、登录邮箱、是否成功、失败原因、客户端 IP、User-Agent、登录时间等信息。用于安全审计和异常登录监控。
- 别名：login-log、auth-log

### 字典管理
- 定义：系统配置数据管理模块，用于维护系统中的静态配置数据。分为字典类型（typeKey/typeName）和字典数据（label/value/sort）两个层次。字典数据按 typeKey 分组，支持启用/禁用状态控制，常用于下拉选项、状态枚举等场景。
- 别名：dict-management、dictionary

### 在线用户
- 定义：系统会话管理功能，用于查看和管理当前登录的用户会话。支持按单个会话或按用户强制下线，同一用户可多端同时登录。该模块采用角色鉴权（要求 admin 角色），而非权限点鉴权，是系统中唯一的例外。
- 别名：online-user、session-management

### Flyway 迁移
- 定义：数据库版本管理工具，用于自动化执行数据库结构变更。迁移脚本位于 `src/main/resources/db/migration/` 目录，采用 V{版本号}__描述.sql 命名规范。支持增量升级和基线初始化，确保不同环境的数据库结构一致性。
- 别名：database-migration、flyway
